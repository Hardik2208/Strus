1. Purpose

The Project Domain is the heart of Strus.

Everything of business value eventually becomes a Project.

Unlike Jira, Notion, ClickUp or Linear, a Project in Strus is not merely a collection of tasks.

A Project represents a legally agreed professional engagement between exactly one client organization and one or more professionals.

It owns:

Contract lifecycle
Participant lifecycle
Invitation lifecycle
Agreement
Execution state
Timeline
Activity
WorkPlan reference
FinancialAccount reference

A Project does not directly own milestones, submissions or payments.

Those belong to their own aggregates.

2. Project Philosophy

A Project answers one question:

"What professional engagement are these people executing together?"

Everything else is attached to that engagement.

22. Business Rules
Every project belongs to exactly one organization.
Every project has exactly one agreement.
Every project has exactly one work plan.
Every project has exactly one financial account.
Exactly one client exists per project.
Multiple contractors are allowed.
Multiple professionals are allowed.
Viewers are read-only.
Contract editing is unrestricted before activation.
Agreement modifications require participant re-review.
Reinvitation after decline requires a 24-hour cooldown.
Projects cannot become ACTIVE until all pending invitations are resolved.
Every approval and invitation response is permanently recorded.
Mutual termination requires explicit approval history.
23. Golden Rules
Project is the contractual aggregate.
Organizations administer projects but do not execute them.
Participants belong to projects, not organizations.
Invitations preserve complete decision history.
Agreements are versioned, never overwritten.
Activation freezes contractual structure.
Execution begins only after project activation.
Approval is explicit, never inferred.
Mutual termination is a business workflow, not a status change.
Every project references—but does not own—execution, finance, and communication aggregates.
Architecture Observation (Important)

While writing this, I found one additional entity that should be added before we freeze the canonical model:

AgreementVersion

Earlier we said "Agreement versions," but we never modeled it as an entity.

It should be:

Project
    │
    ▼
Agreement
    │
    └── AgreementVersion (1..N)

Each version stores:

Changed clauses
Previous values
Changed by
Timestamp
Change reason

This is essential because your workflow requires participants to re-review modified agreements before activation. Without a dedicated AgreementVersion entity, you'd lose the exact version each participant accepted.
