16. Architecture Review (Corrections Before Freezing)

After reviewing everything we've built over these documents, there are three final corrections I would make before freezing the architecture:

1. ApprovalRecord

Right now it's referenced in both the Project and Execution domains.

It should exist only once.

Since approvals happen for milestones, amendments, extensions, and terminations, I recommend introducing a generic:

ApprovalRequest
    │
    └── ApprovalDecision

with:

referenceType
referenceId

This removes duplicate approval tables and gives you a single approval workflow.

2. TimelineEvent and ActivityEvent

These are almost identical structurally.

Instead of two tables, consider:

Event
├── type = TIMELINE
└── type = ACTIVITY

If you anticipate different querying patterns, keep them separate. Otherwise, a unified event model is cleaner.

3. Conversation.referenceType

You've used this pattern multiple times (Conversation, ActionRequest, AuditLog, Notification).

Rather than every table implementing its own polymorphic reference, standardize it:

Reference
├── referenceType
└── referenceId

This creates consistency across the entire platform.
