1. Purpose

The Canonical Domain Model defines the complete business structure of Strus.

Unlike the Backend Architecture documents, which describe engineering principles and system design, this document defines the business entities that exist inside the platform and the relationships between them.

Every future artifact—including the PostgreSQL schema, Prisma models, repositories, services, DTOs, APIs, and frontend state—is derived from this model.

The Canonical Domain Model is the highest authority for business data representation.

2. Design Philosophy

The model is built around five principles.

Business First

Every entity exists because it represents a real business concept.

No entity exists solely because of database convenience.

Explicit Relationships

Every relationship must be directly represented.

Implicit relationships are avoided.

Immutable Business History

Business history is never overwritten.

State transitions create records.

Financial operations create records.

Score changes create records.

Invitation decisions create records.

Revision requests create records.

The platform values traceability over convenience.

Separation of Concerns

Business concepts remain independent.

For example,

Storage is not part of Submission.

Submission is not part of Payment.

Payment is not part of Reputation.

Each domain owns only its own data.

Aggregate Consistency

Each aggregate owns its consistency boundary.

Transactions never span multiple unrelated aggregates unless absolutely required.

8. Golden Rules
Every entity represents a business concept.
Every aggregate has one root.
Aggregate boundaries define transactional consistency.
Child entities are modified only through their aggregate root.
History is preserved through immutable records.
Reputation is calculated, never manually edited.
Storage is completely separated from business data.
Communication is modeled through conversations rather than individual chat implementations.
Financial operations belong to Financial Accounts, not Projects.
The Canonical Domain Model is the source of truth for every future database, API, and implementation artifact.
