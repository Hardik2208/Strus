1. Purpose

The Financial Domain manages every monetary transaction occurring within Strus.

Unlike traditional freelance marketplaces, Strus does not simply record payments.

It manages:

Contract funding
Milestone escrow
Payment releases
Refunds
Platform fees
Settlement history
Financial audit trail

Every movement of money must be completely traceable.

No financial record is ever overwritten or deleted.

2. Financial Philosophy

One of the biggest architectural decisions we made is:

Money never belongs directly to a Project.

Every project owns a dedicated Financial Account.

Project
    │
    ▼
FinancialAccount

Everything financial happens through this account.

24. Business Rules
Every project owns one Financial Account.
Every milestone owns one payment obligation.
Funding rules depend on milestone count.
Escrow always receives money before professionals.
Payments are released milestone-wise.
Multiple professionals split milestone payments.
Platform fee is calculated on total contract value.
Refunds occur only from remaining escrow.
Every financial action creates a Ledger Entry.
Financial history is immutable.
25. Golden Rules
Money never belongs directly to Projects.
Escrow is mandatory.
Ledger is the financial source of truth.
Payments are immutable.
Refunds never overwrite payments.
Platform fee calculation is deterministic.
Funding obligations are separate from transactions.
FinancialAccount is the accounting boundary.
Every gateway interaction is recorded.
Financial history is permanent.
🚨 Architecture Observation (Important Correction)

While reviewing all the business rules, I found one flaw in the current financial model that should be corrected before freezing it.

Earlier we modeled:

Milestone
    │
    ▼
MilestonePayment

However, your business rule says:

If one milestone has multiple professionals, the contractor decides how much each professional receives before the project starts.

That means payment is not actually owned by the milestone.

The correct model is:

Milestone
    │
    ▼
MilestoneFunding
    │
    ├── Allocation A → Professional A
    ├── Allocation B → Professional B
    └── Allocation C → Professional C

So I recommend replacing MilestonePayment with MilestoneFunding, and introducing a child entity:

MilestoneFunding
    │
    └── PaymentAllocation

Where:

MilestoneFunding = how much escrow is reserved for that milestone.
PaymentAllocation = how that funded amount is divided among professionals.

This matches your business rules exactly and avoids ambiguity when one milestone has multiple assigned professionals. It is the last structural improvement I would make before freezing the canonical ER model.
