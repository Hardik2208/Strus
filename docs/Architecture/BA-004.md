1. Purpose

The purpose of this document is to define how dependencies are managed, how requests travel through the backend, how services coordinate complex business operations, and how transactions remain consistent throughout the system.

This document establishes the internal execution model of the Strus backend.

Every HTTP request, WebSocket event, scheduled task, and background job should follow these principles.

2. Dependency Injection Philosophy

Dependency Injection (DI) is used to decouple business logic from implementation details.

Business logic should never instantiate infrastructure objects directly.

Incorrect:

const prisma = new PrismaClient();
const projectRepo = new ProjectRepository(prisma);

Correct:

ProjectService

↓

IProjectRepository

↓

ProjectRepository

↓

Prisma

The service depends on an interface, not an implementation.

3. Why Dependency Injection

Dependency Injection provides:

Loose coupling
Easier testing
Swappable implementations
Better maintainability
Clear ownership
Future service extraction

Every service should remain unaware of how data is stored.

31. Performance Principles

Within a service:

Minimize database queries.
Batch related operations.
Avoid N+1 queries.
Fetch only required fields.
Use transactions wisely.
Never perform expensive work inside database transactions if it can happen afterward.
32. Anti-Patterns to Avoid

Never:

Instantiate repositories inside services.
Access Prisma directly from controllers.
Call another module's repository.
Publish events before commit.
Mix business validation with HTTP validation.
Store request state inside services.
Depend on singleton mutable state.

35. Golden Rules

Every backend engineer working on Strus must follow these principles:

Services depend on interfaces, never implementations.
One business operation owns one transaction.
Events are emitted only after successful commits.
Business logic is transport-independent.
Validation occurs in three stages: request, business, database.
Controllers orchestrate requests; services own decisions.
External providers are accessed only through adapters.
Concurrency is handled by the backend, not the frontend.
Critical operations are idempotent.
Every request is traceable through correlation IDs.
Summary

This document defines the execution model of the Strus backend. It establishes dependency injection, request processing, service orchestration, transaction ownership, event publishing, concurrency control, and resilience strategies. By following these rules, every business operation remains deterministic, testable, and scalable, while keeping domain logic independent of frameworks and infrastructure.
