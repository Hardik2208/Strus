1. Purpose

The Repository Layer isolates business logic from the persistence mechanism.

Services should never know:

PostgreSQL
Prisma
SQL
Indexes
Transactions
Query optimization

Their only responsibility is business execution.

The Repository Layer owns every interaction with persistent storage.

2. Repository Philosophy

Repositories answer one question:

How is data retrieved or persisted?

They never answer:

Should this business operation happen?

Business decisions belong to Services.

Persistence belongs to Repositories.

3. Repository Architecture
Controller

↓

Service

↓

Repository Interface

↓

Repository Implementation

↓

Prisma

↓

PostgreSQL

Repositories are the only gateway to the database.

4. Repository Responsibilities

A repository may:

Query data
Create records
Update records
Delete (soft delete where applicable)
Execute transactions
Build optimized queries

A repository must never:

Check permissions
Validate business rules
Send notifications
Calculate scores
Publish events
5. Interface-Driven Design

Every repository has an interface.

Example

IProjectRepository

IMilestoneRepository

IAgreementRepository

IPaymentRepository

Services depend on interfaces.

Not implementations.

6. One Repository Per Aggregate

Avoid repositories that manage multiple domains.

Correct

ProjectRepository

MilestoneRepository

SubmissionRepository

Incorrect

DatabaseRepository

CommonRepository

AppRepository

Repositories should align with aggregates.

7. Repository Structure
repositories/

project.repository.ts

milestone.repository.ts

submission.repository.ts

artifact.repository.ts

payment.repository.ts

Simple.

Predictable.

8. Prisma Client

Exactly one Prisma Client instance exists.

Application

↓

Prisma Client

↓

Repositories

Never instantiate Prisma inside repositories.

9. Query Philosophy

Repositories should query only required fields.

Avoid

SELECT *

Prefer

SELECT
id,
status,
createdAt

Reducing transferred data improves performance.

10. Read Operations

Repositories expose business-oriented queries.

Example

Instead of

find()

Prefer

findProjectById()

findActiveProjects()

findProjectsByOrganization()

findProjectParticipants()

Method names describe intent.

11. Write Operations

Every write should be explicit.

Example

createProject()

archiveProject()

updateDeadline()

approveSubmission()

Avoid generic methods like

save()

They obscure intent.

12. Transactions

Repositories participate in transactions but do not own business workflows.

Example

Project Service

↓

Transaction

↓

Project Repository

↓

Milestone Repository

↓

Participant Repository

↓

Commit

Repositories simply execute within the provided transaction context.

13. Query Optimization

Repositories own optimization.

Examples:

Select only required columns.
Use pagination.
Batch related queries.
Avoid N+1 problems.
Use joins appropriately.
Respect indexes.

Services remain unaware of these details.

14. Pagination Strategy

Large datasets use cursor pagination.

Example

findProjects({
cursor,
limit
})

Never expose offset pagination for high-volume endpoints.

15. Soft Deletes

Repositories handle soft deletion.

Instead of deleting:

DELETE

Update

deletedAt

Queries exclude deleted records by default.

Administrative queries may opt in.

16. Prisma Transactions

Business services initiate transactions.

Repositories receive the transaction client.

Service

↓

Prisma Transaction

↓

Repository A

Repository B

Repository C

This ensures one atomic operation.

17. Eager vs Lazy Loading

Repositories choose the appropriate loading strategy.

Use eager loading for:

Participants
Milestones
Organization

Use separate queries when relationships are large or optional.

18. Caching

Repositories do not manage cache directly.

Caching belongs to dedicated cache services.

Repository responsibility ends with data retrieval.

19. Error Translation

Repositories translate database-specific errors into domain-neutral exceptions.

Example

Unique constraint violation

↓

DuplicateEmailError

The service layer should never receive raw Prisma errors.

20. Audit Awareness

Repositories never create audit logs directly.

Audit generation is orchestrated by the Service Layer after successful business operations.

21. Bulk Operations

Support efficient batch processing.

Examples:

Bulk notification creation.
Bulk artifact metadata updates.
Bulk participant insertion.

Avoid row-by-row database writes where possible.

22. Query Monitoring

Repositories should expose metrics:

Query duration.
Rows returned.
Slow query detection.
Transaction duration.

These feed the observability system.

23. Repository Testing

Repository tests verify:

SQL correctness.
Constraints.
Index usage.
Transaction behavior.
Prisma mappings.

Business rules are tested elsewhere.

24. Golden Rules
Repositories own persistence.
Services own business logic.
Controllers own HTTP.
Prisma is hidden behind repositories.
Queries are optimized.
Transactions are respected.
Interfaces abstract implementations.
Repository methods express business intent.
25. Definition of a Well-Designed Repository

A repository is considered complete when:

It encapsulates all persistence logic for its aggregate.
It exposes business-oriented query methods.
It hides Prisma implementation details.
It supports efficient querying and transactions.
It contains no business rules.
It can be replaced without affecting services.
Summary

The Repository Layer provides a clean abstraction between business logic and PostgreSQL. It centralizes persistence, query optimization, transaction participation, and Prisma integration while remaining completely free of business decisions. This separation ensures that the Service Layer stays focused on domain behavior and that persistence can evolve independently.
