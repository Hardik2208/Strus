1. Purpose

Security is the foundation of Strus.

Every contract, payment, approval, dispute, and reputation score depends on the identity of the actor performing an action. The backend must therefore treat authentication and authorization as first-class architectural concerns rather than middleware.

The objective of this document is to define how users prove their identity, how permissions are evaluated, how sessions are managed, and how every request is secured.

Security decisions should always favor correctness over convenience.

2. Security Principles

Every security implementation in Strus follows these principles.

Authentication before Authorization.
Never trust frontend state.
Every request is independently validated.
Least privilege by default.
All sensitive operations are auditable.
Credentials are never stored in plaintext.
Sessions are revocable.
APIs remain stateless.
Permissions are evaluated dynamically.
Security must not depend on hidden UI elements.

37. Golden Rules
Authentication proves identity.
Authorization determines capability.
Permissions are never stored inside JWTs.
Refresh Tokens are always rotated.
OTPs never reach PostgreSQL.
Every request is independently authorized.
Security middleware executes before business logic.
Every sensitive action is audited.
Never trust client-side validation.
All permissions are state-aware.
38. Definition of a Secure Backend

A secure backend:

Rejects unauthorized requests before business execution.
Stores no plaintext secrets.
Maintains revocable sessions.
Supports fine-grained permissions.
Audits every sensitive operation.
Prevents privilege escalation.
Protects against common web vulnerabilities.
Maintains complete traceability for identity and access events.
Summary

The Strus security architecture separates authentication, authorization, verification, and permission evaluation into independent layers. JWTs provide stateless authentication, refresh tokens maintain secure long-lived sessions, and authorization is determined dynamically based on organization roles, project roles, and business state rather than static claims. Every request passes through a consistent security pipeline, and every sensitive operation is auditable, ensuring that the platform remains secure, scalable, and maintainable as it evolves.
