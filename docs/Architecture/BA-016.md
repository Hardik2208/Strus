1. Purpose

Software is not considered complete when it compiles.

Software is complete only when its behavior is verified.

Testing in Strus exists to guarantee that business rules remain correct as the platform evolves.

The objective is not merely achieving high code coverage, but ensuring that contractual workflows, financial operations, permissions, and business invariants remain correct under every supported scenario.

Testing is treated as an engineering discipline rather than a release-phase activity.

Examples:

Worker + Queue

8. Business Rule Testing

Every business rule requires dedicated tests.

Examples:

Milestone payment must exist before activation.
Revision limit enforced.
Deadline extension requires approval.

Every bug fixed receives a regression test.

Rule:

34. Definition of a Reliable System

A reliable Strus backend should:

Preserve business invariants.
Detect regressions before deployment.
Remain resilient under infrastructure failures.
Maintain API compatibility.
Guarantee financial correctness.
Validate every permission path.
Scale without introducing regressions.
Inspire confidence through automation rather than manual testing.
Summary

The testing strategy of Strus prioritizes business correctness over raw coverage. Unit tests verify business rules, integration tests validate collaboration between components, and end-to-end tests exercise complete contractual workflows. Performance, chaos, and regression testing ensure that the platform remains reliable as it evolves, while CI enforces quality before every deployment. The result is a backend where confidence comes from automated verification rather than assumptions.
