1. Purpose

The database is the most valuable asset of Strus.

Business logic can be rewritten.

APIs can change.

Frontend can be redesigned.

Infrastructure can migrate.

The database cannot be redesigned every six months.

Therefore every entity, relationship, index and constraint must be designed assuming:

Millions of users
Millions of projects
Tens of millions of milestones
Hundreds of millions of notifications
Billions of audit records

The database is designed around business concepts, not UI requirements.

2. Database Philosophy

Strus uses PostgreSQL as the primary datastore.

The database is responsible for:

Data integrity
Referential integrity
Transactions
Constraints
Efficient querying

The database is not responsible for:

Business logic
Permission evaluation
Notification delivery
AI processing

Those belong to the application layer.

3. Source of Truth

PostgreSQL is the only source of truth.

Redis

Cache only

Cloudinary

File storage only

WebSocket

Communication only

Frontend

Presentation only

Business state exists only in PostgreSQL.

4. Data Modeling Principles

Every table should satisfy:

Single responsibility
Strong normalization
Explicit ownership
Immutable history
Minimal redundancy

Every table should answer one business question.

Example:

User

Who is using Strus?
Project

What work is being executed?
Invitation

Who has been invited?

Never mix unrelated responsibilities.

5. Primary Key Strategy

Every entity uses

UUID

Reasons

Distributed systems
No ID enumeration
Better API security
Easier future sharding

Never use auto-increment IDs.

6. Foreign Keys

Every relationship must use foreign keys.

Never store

projectName

inside Milestone.

Instead

projectId

Foreign keys guarantee consistency.

Never create indexes "just in case."

Every index should support a frequent access pattern.

20. Cascade Rules

Delete should be explicit.

Never use

ON DELETE CASCADE

blindly.

Business entities should usually be archived instead of deleted.

Recommended

RESTRICT

SET NULL

depending on the relationship.

21. Transactions

Examples

Milestone Approval
Payment Release

Never split one business event across multiple transactions.

Select only required columns.

28. Search

Future

30. Backup Strategy

Daily backups.

Point-in-time recovery enabled.

Weekly snapshot verification.

Monthly restore testing.

A backup is only useful if restoration is verified.

31. Migration Philosophy

Reviewed

33. Database Evolution

Schema changes should preserve compatibility.

Never break existing APIs without versioning.

Prefer additive migrations.

34. Performance Monitoring

Monitor

35. Golden Rules
PostgreSQL is the source of truth.
Every entity has one responsibility.
Normalize business data.
Version mutable business objects.
Keep immutable history.
Use UUIDs everywhere.
Index based on query patterns.
Use JSONB only for variable provider metadata.
Never store files inside PostgreSQL.
Design every table assuming millions of rows.

37. Definition of a Good Database

A well-designed Strus database should:

Preserve every business rule through constraints and relationships.
Support transactional consistency across complex workflows.
Scale without redesign as data volume grows.
Maintain complete historical traceability.
Minimize duplication while allowing efficient queries.
Remain understandable to new engineers through consistent naming and structure.
Allow future services, AI, analytics, and integrations without changing the core schema.
Summary

This document establishes the persistence strategy for Strus. It defines how data is modeled, stored, related, indexed, versioned, and retained. The database is treated as a long-term business asset whose design prioritizes integrity, auditability, scalability, and maintainability over short-term convenience.
