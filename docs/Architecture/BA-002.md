1. Purpose

This document defines the overall architecture of the Strus backend system.

Unlike the previous document, which established engineering philosophy, this document explains how the entire system is organized, how requests flow through it, and how every major component interacts.

The objective is to ensure that every engineer working on Strus understands the system before writing a single feature.

2. High-Level System Vision

Strus is designed as a Modular Monolith.

Every feature exists as an independent module while sharing a single runtime, database, and deployment.

This provides:

low operational complexity
fast development velocity
simple deployments
transactional consistency
clear module boundaries

without introducing the operational burden of microservices.

The architecture intentionally prepares every module for future extraction if independent scaling becomes necessary.

24. Core Architectural Principles

Every engineering decision must satisfy these principles:

Business logic is isolated from transport and infrastructure.
Domain modules are independent and cohesive.
All critical operations are transactional.
External integrations are accessed through adapters.
State transitions are explicit and validated.
PostgreSQL is the authoritative source of truth.
Redis is disposable.
Cloudinary stores files, not business data.
Controllers remain thin.
Services encapsulate business rules.
Repositories encapsulate persistence.
Modules communicate through services or domain events.
The system remains horizontally scalable.
Summary

This document defines how the Strus backend is organized as a system. It establishes the architectural layers, request lifecycle, module boundaries, infrastructure responsibilities, communication patterns, transaction philosophy, and deployment model.

Every future backend component should conform to this architecture to ensure consistency, maintainability, and scalability as Strus grows from a modular monolith into a platform capable of supporting enterprise-scale workloads.
