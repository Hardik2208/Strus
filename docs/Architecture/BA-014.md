1. Purpose

Performance is not an optimization added after the product succeeds.

It is an architectural characteristic designed from the beginning.

The goal of the Strus backend is not simply to be "fast." It must remain predictable under increasing load while preserving business correctness.

The platform should comfortably support:

Millions of users
Hundreds of thousands of concurrent projects
Millions of milestones
Thousands of simultaneous WebSocket connections
High read and write throughput

without redesigning the core architecture.

2. Scalability Philosophy

Never optimize prematurely.

Never ignore scalability completely.

Follow this progression:

Correctness

↓

Maintainability

↓

Observability

↓

Performance

↓

Scalability

Performance never comes before correctness.

Optimize by:

Proper indexes

Cursor pagination
Connection pooling

8. Redis Philosophy

Redis is an optimization layer.

Never the source of truth.

If Redis disappears,

the application must still function.

34. Golden Rules
PostgreSQL is optimized before Redis.
Cache accelerates; it never defines truth.
Backend instances remain stateless.
Scale horizontally, not vertically.
Use cursor pagination.
Keep requests lightweight.
Push expensive work to background workers.
Measure everything.
Optimize based on evidence.
Design for growth, not today's traffic.

36. Definition of a Scalable System

A scalable Strus backend should:

Maintain predictable latency under increasing load.
Scale horizontally without architectural changes.
Keep business state centralized in PostgreSQL.
Treat Redis as an optimization layer.
Offload expensive operations to workers.
Support future read replicas and service extraction.
Remain observable and measurable at every layer.
Summary

The performance architecture of Strus is built around stateless application servers, optimized PostgreSQL access, Redis as a transient acceleration layer, and horizontally scalable background workers. Caching, connection pooling, efficient querying, and direct-to-cloud uploads ensure low latency while preserving correctness. Rather than optimizing prematurely, the system is designed to evolve from a modular monolith into a distributed platform only when real scaling requirements justify the added complexity.
