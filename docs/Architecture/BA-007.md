1. Purpose

The API layer is the public contract between the frontend and the backend.

It must remain:

Stable
Predictable
Versioned
Secure
Consistent

APIs should expose business capabilities rather than database tables.

The frontend should never need to understand the internal implementation of the backend.

2. API Design Philosophy

Every endpoint should represent a business action.

Good

POST /projects
POST /projects/{id}/start
POST /milestones/{id}/submit

Bad

POST /createProject
POST /updateProjectStatus
POST /setMilestoneCompleted

The URL represents the resource.

The HTTP verb represents the action.

3. REST Principles

Every API follows REST conventions.

GET

Read
POST

Create / Business Action
PUT

Replace Entire Resource
PATCH

Partial Update
DELETE

Soft Delete / Archive

Avoid using POST for simple updates.

4. API Versioning

Every endpoint begins with

/api/v1/

Example

/api/v1/projects
/api/v1/auth/login

Never expose unversioned APIs.

Future

/api/v2/

without breaking old clients.

5. URL Naming

Use plural nouns.

Correct

/users
/projects
/milestones

Avoid

/getProjects
/projectList
/createProject
6. Nested Resources

Represent ownership.

Example

/projects/{projectId}/milestones
/projects/{projectId}/participants
/milestones/{id}/submissions

Do not over-nest.

Maximum depth:

3
7. Resource IDs

Always UUIDs.

Example

/projects/7d67d98d...

Never sequential integers.

10. Response Philosophy

Every successful response follows one format.

{
    "success": true,
    "data": {},
    "meta": {},
    "message": ""
}

Every error follows one format.

{
    "success": false,
    "error": {
        "code": "",
        "message": "",
        "details": []
    }
}

Consistency reduces frontend complexity.

11. HTTP Status Codes

Only use standard codes.

Successful read.

201 Created

204 No Content

400 Bad Request

401 Unauthorized

403 Forbidden

Permission denied.

404 Not Found

Resource missing.

409 Conflict

Business state conflict.

422 Unprocessable Entity

Business validation failure.

500 Internal Server Error

Unexpected failure.

33. Golden Rules
APIs expose business capabilities.
URLs represent resources.
HTTP verbs represent actions.
Validation occurs before business logic.
Responses follow one standard format.
Errors remain predictable.
APIs are versioned from day one.
Pagination is mandatory.
Business actions use dedicated endpoints.
Documentation is generated automatically.
34. Definition of a Well-Designed API

A well-designed API should:

Be intuitive without documentation.
Maintain backward compatibility.
Minimize frontend complexity.
Clearly represent business operations.
Support future expansion without breaking existing clients.
Remain consistent across every module.
Be secure, observable, and testable.
Summary

The Strus API architecture is designed around business operations rather than CRUD. Every endpoint follows consistent naming, validation, versioning, authentication, authorization, and response standards. REST is used to mutate and retrieve business state, while WebSockets synchronize changes in real time. This ensures that the API remains stable, predictable, and scalable as the platform evolves.
