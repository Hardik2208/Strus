1. Purpose

Strus is a collaborative execution platform where multiple participants work on the same contract simultaneously.

Polling the backend every few seconds would:

Increase infrastructure cost
Introduce unnecessary latency
Create inconsistent UI states
Scale poorly

Instead, Strus uses WebSockets for real-time synchronization while REST remains responsible for business operations.

This document defines the communication architecture of the platform.

2. Communication Philosophy

Every communication mechanism has exactly one responsibility.

Technology	Responsibility
REST API	Execute business operations
WebSocket	Synchronize business state
Notifications	Inform users
Email	Long-form communication
Timeline	Immutable project history
Activity Feed	Organization activity

Never mix responsibilities.

REST changes state.

36. Definition of a Well-Designed Communication System

The communication layer should:

Deliver updates in real time.
Never compromise business consistency.
Recover gracefully after disconnects.
Support multiple delivery channels.
Scale horizontally.
Keep transient data outside PostgreSQL where appropriate.
Preserve immutable business history.
Maintain strict authorization over every communication channel.
Summary

The communication architecture of Strus separates business execution from real-time synchronization. REST APIs perform all state-changing operations, while WebSockets deliver live updates after successful commits. Notifications inform users of actionable events, timelines preserve immutable project history, and chats remain scoped to projects, revisions, and disputes. This layered design ensures responsive collaboration without sacrificing consistency, scalability, or security.
