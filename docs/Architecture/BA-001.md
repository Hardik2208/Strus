1. Introduction

Strus is a professional project execution platform designed to replace fragmented workflows used by clients and professionals during project delivery.

It is not a freelancer marketplace. User discovery happens outside Strus. The platform begins once both parties decide to work together.

The primary objective of the backend is not simply exposing REST APIs. It is responsible for preserving business integrity, enforcing contractual workflows, maintaining auditability, and ensuring that every business operation remains deterministic under concurrent access.

Every architectural decision should prioritize correctness over convenience.

The backend must remain maintainable as the product evolves from an MVP into a multi-service platform supporting millions of projects.

2. Engineering Philosophy

The backend follows six core engineering principles.

2.1 Business Logic First

Business rules define the architecture.

Controllers, APIs, database schema, and services exist only to implement approved business rules.

Example:

Incorrect approach

Create endpoint
↓
Write logic
↓
Think about rules later

Correct approach

Business Rule
↓
Domain Model
↓
Service
↓
API

No business logic should exist without an associated business rule.

2.2 Domain Driven Thinking

The backend is designed around business domains instead of technical layers.

Examples of domains include:

Authentication
Organizations
Projects
Invitations
Milestones
Payments
Submissions
Disputes
Notifications

Every domain owns its own entities, services, repositories, validations, and APIs.

Cross-domain communication must remain minimal.

2.3 Simplicity Over Cleverness

Solutions should optimize for readability rather than clever implementation.

Every engineer joining the project should understand the architecture within days rather than weeks.

Avoid:

Deep inheritance
Hidden magic
Dynamic runtime behaviors
Complex abstractions without measurable benefit

Prefer explicit implementations.

2.4 Long-Term Maintainability

Every feature must be designed assuming it will continue evolving over the next five years.

The architecture must support:

New payment providers
New authentication providers
New contract types
AI integrations
Multiple storage providers
Multiple notification channels
Distributed services

without requiring major redesign.

2.5 Security by Design

Security is not an afterthought.

Every module must assume hostile input.

Every request must pass through validation, authorization, and auditing before modifying business data.

Sensitive information should never be trusted based solely on frontend behavior.

Authorization is enforced entirely by the backend.

2.6 Explicit Business State

Every business entity must exist in a well-defined state.

Examples:

Project

DRAFT
↓
PENDING_ACCEPTANCE
↓
ACTIVE
↓
ON_HOLD
↓
COMPLETED

Invitation

PENDING
↓
ACCEPTED

or

DECLINED

or

EXPIRED

Implicit states are prohibited.

Every state transition must be intentional and validated.

The backend is responsible for preserving project integrity.

The platform guarantees:

structured collaboration

revision tracking

audit history

rather than merely storing project data.

4. Architectural Goals

The backend has five primary objectives.

Scalability

The architecture must support:

millions of users
millions of projects
concurrent milestone submissions
thousands of active websocket connections
high write throughput

Horizontal scaling must remain possible.

Reliability

Business operations should never leave inconsistent state.

Examples:

Either

Payment succeeds
Milestone updates
Timeline updates
Notification sent

or

Nothing changes.

Partial success is unacceptable.

Maintainability

Every module should evolve independently.

Adding a new payment provider should not require changes inside milestone logic.

Adding a new contract type should not require controller modifications.

Observability

Every important business event should be traceable.

Questions such as:

Who approved?

Who edited?

When?

Why?

should always have deterministic answers.

Extensibility

Future features should integrate without redesigning the existing architecture.

Examples include:

AI
GitHub integrations
Slack integrations
Calendar integrations
Digital signatures
Enterprise organizations

9. Architecture Evolution Strategy

The backend should evolve in stages.

Stage 1 — Modular Monolith (Current)

One backend application with clear module boundaries.

Shared database.

Shared deployment.

Stage 2 — Internal Service Separation

Introduce asynchronous jobs and internal event processing.

Stage 3 — Selective Microservices

Extract only domains that benefit from independent scaling, such as:

Notifications
Payments
AI Processing
Search
Analytics

The goal is not to become a microservice architecture prematurely. A well-structured modular monolith will provide lower operational complexity while allowing future extraction of services when justified by scale.

10. Definition of Success

The architecture is considered successful if:

New engineers can understand it quickly.
Business rules are implemented consistently.
New features require minimal changes to existing modules.
Critical operations remain transactional.
The system scales horizontally when required.
Every important business action is auditable.
Domain boundaries remain clean over time.
Summary

This document establishes the philosophy behind Strus. It defines how engineering decisions are made, not how individual modules are implemented. Every subsequent architecture document should align with these principles, ensuring that the backend remains consistent, maintainable, and scalable as the platform grows.
