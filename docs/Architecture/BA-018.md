1. Purpose

This document defines how engineering is practiced within Strus.

It is not about architecture.

It is about engineering culture.

Every future engineer joining the project should be able to understand:

How code is written
How features are designed
How reviews are performed
How architecture evolves
How technical debt is managed

This document serves as the engineering handbook for the project.

2. Engineering Philosophy

Strus follows one guiding principle:

Build software that is easy to change.

Every engineering decision should optimize for:

Readability
Predictability
Maintainability
Correctness

before

Cleverness
Premature optimization
3. Core Principles

Every engineer must follow these principles:

Business first.
Simplicity over complexity.
Explicit over implicit.
Composition over inheritance.
Interfaces over implementations.
Small modules over large services.
Business rules over framework conventions.
Deterministic behavior over convenience.
4. Code Quality Standards

Every pull request should satisfy:

✓ Builds successfully

✓ Passes all tests

✓ No lint errors

✓ No TypeScript errors

✓ Documentation updated

✓ Business rules verified

Code quality is non-negotiable.

5. SOLID Principles

The backend follows SOLID.

Single Responsibility

Every class should have one reason to change.

Open/Closed

Prefer extension.

Avoid modification.

Liskov

Derived implementations should behave consistently.

Interface Segregation

Small interfaces.

Never giant interfaces.

Dependency Inversion

Depend on abstractions.

Never implementations.

6. DRY

Avoid duplicate business logic.

Shared behavior belongs in:

Domain Services
Shared Utilities
Policies

Not copy-paste.

7. KISS

Simple solutions outperform clever solutions.

If two implementations solve the same problem,

prefer the simpler one.

8. YAGNI

Never build features

"just in case."

Only implement requirements supported by business value.

Future-proof architecture.

Not future-proof code.

9. Code Organization

Modules own:

Controllers
Services
Repositories
DTOs
Validators
Policies
Events

Cross-module leakage is prohibited.

10. Naming

Names should describe business intent.

Good

ApproveMilestoneService

Bad

DataProcessor
11. Function Design

Functions should:

Perform one task.
Be deterministic.
Have minimal side effects.
Return predictable results.

Large functions should be decomposed.

12. Class Design

Avoid "God Objects."

Example

Avoid

ProjectService

2,000 lines.

Prefer

CreateProjectService

StartProjectService

ArchiveProjectService
13. Comments

Code should explain how.

Comments should explain why.

Never comment obvious code.

Wrong

// Increment i
i++;

Right

// Required to preserve milestone ordering after reassignment
14. Error Handling

Never swallow exceptions.

Always

Handle
Log
Translate
Re-throw when appropriate

Every error should have context.

Is it actively maintained?
Is it widely adopted?
Can we build it ourselves easily?
Is it worth the complexity?

35. Definition of Engineering Excellence

Engineering excellence is achieved when:

Business rules are faithfully implemented.
Architecture remains understandable after years of development.
Features can be added without rewriting existing systems.
Failures are observable and recoverable.
Code reviews improve quality rather than block progress.
Documentation stays synchronized with implementation.
New engineers become productive quickly.
Users trust the platform because the system behaves consistently.
Final Summary

The Strus backend architecture is built around business-first engineering. Across these eighteen documents, the system has been defined from philosophy through implementation, covering domain modeling, module organization, services, repositories, APIs, persistence, events, payments, storage, communication, observability, testing, deployment, and engineering practices.

The result is a modular, event-driven architecture designed to evolve from a high-quality modular monolith into a scalable distributed platform without sacrificing maintainability or correctness. Every major engineering decision—from how a request is processed to how money is handled, files are retained, events are published, and deployments are executed—follows consistent principles rooted in explicit business rules, transactional integrity, and long-term extensibility.

This architecture serves not only as implementation guidance but also as the engineering handbook for Strus, ensuring that future contributors can build upon a shared foundation while preserving the platform's reliability, scalability, and product vision.a
