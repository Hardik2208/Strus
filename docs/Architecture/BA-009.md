1. Purpose

As Strus grows, a single business operation often triggers many independent tasks.

For example, approving a milestone should not only update the milestone status. It should also:

Release payment
Notify professionals
Notify the client
Create timeline entries
Update activity feeds
Recalculate reputation
Broadcast WebSocket events
Trigger analytics

If all of this is executed synchronously, request latency increases, coupling grows, and failures become difficult to isolate.

This document defines how Strus uses domain events, background workers, and real-time communication to keep the system responsive, scalable, and maintainable.

2. Event-Driven Philosophy

A business operation should focus only on its primary responsibility.

Example:

Approve Milestone

↓

Approve Database State

↓

Commit Transaction

↓

Publish Event

↓

Everything else happens independently

The service should not know who listens to the event.

7. Event Ownership

Only the module that owns a business action may publish its event.

Example:

Payment Module publishes:

Never publish another module's events.

28. Failure Isolation

Worker failures must never affect committed business state.

Example:

↓

↓

Result:

Milestone remains approved.

Email retries independently.

30. Golden Rules
Publish events only after successful commits.
Events describe facts, not commands.
Workers are idempotent.
Side effects never block user requests.
REST mutates state; WebSockets synchronize state.
Workers own asynchronous processing.
Every event carries a Correlation ID.
Timeline and Activity remain immutable.
Failed workers never roll back committed business state.
Background jobs must be independently scalable.
31. Definition of a Well-Designed Event System

A well-designed event architecture should:

Decouple modules.
Reduce request latency.
Support horizontal scaling.
Isolate failures.
Maintain complete traceability.
Allow new consumers without changing producers.
Keep business transactions short.
Ensure every side effect is recoverable.
Summary

The Strus event architecture separates transactional business logic from asynchronous side effects. Business services commit state changes first, then publish domain events that drive notifications, payments, activity feeds, reputation updates, analytics, and real-time synchronization. Workers remain idempotent, independently scalable, and failure-isolated, allowing the platform to grow without coupling unrelated concerns or increasing request latency.
