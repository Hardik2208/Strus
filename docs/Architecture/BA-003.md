1. Purpose

This document defines how every backend feature inside Strus should be organized.

The objective is to ensure that every module:

follows identical architecture
remains independently maintainable
avoids circular dependencies
minimizes coupling
maximizes cohesion

No feature should introduce a new architectural pattern.

Every module must follow the same template.

2. Architectural Philosophy

Strus follows a Domain-Based Modular Monolith.

The project is not organized by technical layers.

Avoid:

controllers/
services/
repositories/
models/
routes/

This becomes unmaintainable as the application grows.

Instead organize around domains.

Authentication

Projects

Organizations

Milestones

Payments

Submissions

Every module owns everything related to its business domain.

32. Future Scalability

Every module should be extractable into a microservice.

Requirements

No circular dependencies
Minimal shared state
Event-driven communication
Clear API boundaries

If extraction requires rewriting business logic, the module boundary was designed incorrectly.

33. Golden Rules

Every engineer working on Strus must follow these rules:

One module = one business domain.
Controllers are thin.
Services contain business logic.
Repositories only access data.
Validation happens before services.
Permissions are module-owned.
Modules communicate through services or events.
Shared code never contains business rules.
Every module has a single public entry point.
Business logic must never depend on Fastify, Prisma, Redis, or Cloudinary.
34. Definition of a Well-Designed Module

A module is considered complete when:

It has a single responsibility.
It owns its entities.
It exposes a clear public interface.
It hides internal implementation.
It can evolve independently.
It has no circular dependencies.
It follows the standard folder structure.
It satisfies all business rules defined for its domain.
Summary

The Strus backend is organized around business domains, not technical layers. Every module follows an identical internal structure, ensuring consistency across the codebase. Controllers remain lightweight, services encapsulate business logic, repositories isolate persistence, and modules communicate only through well-defined interfaces or domain events. This architecture enables independent evolution of features, simplifies testing, and prepares the modular monolith for future service extraction without requiring architectural rewrites.
