# DDS-000 — Database Design Specification Overview

Version: 1.0

Status: Final

Owner: Database Architecture

---

# 1. Purpose

This document defines the database design principles, conventions, and implementation standards used throughout Strus.

Unlike the Business Architecture and Canonical Domain Model, which describe business concepts and aggregate boundaries, this specification defines how those concepts are represented inside PostgreSQL.

Every subsequent Database Design Specification (DDS) document inherits the rules defined here.

This document is the source of truth for:

- PostgreSQL schema design
- Table design
- Relationships
- Constraints
- Indexes
- Enums
- Versioning
- Soft deletion
- Historical data
- Performance strategy
- Migration strategy

No database implementation should contradict this specification.

---

# 2. Scope

This specification covers every persistent entity within Strus.

The database includes the following domains:

- Identity
- Organization
- Project
- Execution
- Submission
- Financial
- Communication
- Dispute
- Reputation
- Infrastructure

Every table belongs to exactly one domain.

Every table belongs to exactly one aggregate.

---

# 3. Design Philosophy

The Strus database is designed as a long-term business asset.

Business logic may evolve.

Services may change.

APIs may be redesigned.

Frontend implementations may be replaced.

The database must remain stable.

Therefore the schema prioritizes:

- Data integrity
- Auditability
- Predictable relationships
- Normalization
- Long-term maintainability
- Migration safety
- Performance
- Future scalability

The database is designed for correctness before convenience.

# 6. Database Philosophy

The PostgreSQL database is the only source of truth.

Redis is a cache.

Cloudinary stores files.

WebSockets distribute events.

Background workers execute asynchronous tasks.

Business state exists only in PostgreSQL.

---

# 7. Database Goals

The schema must support:

Millions of users

Millions of organizations

Millions of projects

Hundreds of millions of milestones

Billions of audit records

Large financial datasets

Long-term historical data

Enterprise customers

Future microservices

Future analytics

Future AI processing

No redesign should be required to support growth.

---

# 8. Normalization Strategy

Business entities follow Third Normal Form (3NF).

Business data is never duplicated.

Relationships are represented using foreign keys.

Denormalization is permitted only when:

- Query performance requires it.
- Dashboard aggregation requires it.
- Search requires it.
- Cached counters improve performance.

Every denormalized field must be documented.

---

# 9. Aggregate Ownership

Every table belongs to exactly one aggregate.

Aggregate roots control all mutations.

Child entities are never modified independently.

Foreign keys must respect aggregate boundaries.

Cross-domain communication occurs through references and domain events.

# 11. Primary Key Strategy

Every business entity uses UUID.

UUIDs prevent:

Predictable identifiers

Distributed ID collisions

Future sharding issues

Auto-increment IDs are prohibited.

# 13. Timestamp Strategy

Every mutable entity contains:

createdAt

updatedAt

Soft-deletable entities additionally contain:

deletedAt

Immutable entities never require updatedAt after creation unless operational metadata changes.

All timestamps use UTC.

---

# 14. Soft Delete Strategy

Soft deletion is used only when business recovery is meaningful.

Typical examples:

Users

Organizations

Projects

StorageObjects

Business records that represent historical events are immutable and never soft deleted.

Examples:

Payments

Ledger Entries

Audit Logs

Performance Records

Invitation Decisions

Settlement Records

---

# 15. Versioning Strategy

Mutable business objects are versioned.

Examples include:

Agreement

WorkPlan

Submission

Future algorithm versions

Previous versions are never overwritten.

Version history remains permanently available.

---

# 16. Immutable Data Strategy

The following entities are append-only.

Audit Logs

Ledger Entries

Performance Records

Invitation Decisions

Timeline Events

Activity Events

Submission History

Gateway Transactions

Refunds

Settlements

History must never be rewritten.

---

# 17. Relationship Strategy

One-to-One

Shared ownership

One-to-Many

Primary relationship pattern

Many-to-Many

Represented using explicit bridge entities

Implicit many-to-many relationships are prohibited.

---

# 18. Constraint Strategy

Every business relationship must be enforced by:

Primary Keys

Foreign Keys

Unique Constraints

Check Constraints

Default Values

Application logic supplements database constraints but never replaces referential integrity.

---

# 19. Index Strategy

Indexes exist only to support query patterns.

Every index must have a documented purpose.

Index categories include:

Primary

Unique

Secondary

Composite

Partial

GIN

Future BRIN where appropriate

Unused indexes should not exist.

---

# 20. JSONB Strategy

JSONB is used only for variable external metadata.

Approved examples:

Cloudinary metadata

Gateway provider payloads

Audit snapshots

Webhook payloads

Rejected examples:

Projects

Users

Milestones

Payments

Core business entities remain relational.

---

# 21. Financial Design Principles

Financial data is immutable.

Ledger Entries represent the source of truth.

Balances are derived.

Gateway state and business state remain independent.

All monetary calculations use fixed precision decimals.

Money is never stored using floating point values.

---

# 22. Storage Design Principles

Files never enter PostgreSQL.

Cloudinary stores physical objects.

PostgreSQL stores metadata.

Artifacts remain permanently.

StorageObjects may expire after retention.

Business history remains available after storage cleanup.

---

# 23. Query Philosophy

Repositories select only required columns.

SELECT * is prohibited.

Pagination is mandatory.

Cursor pagination is preferred.

Offset pagination is reserved for administrative views.

---

# 24. Partitioning Strategy

Initial deployment:

No partitioning.

Future candidates:

Audit Logs

Timeline Events

Activity Events

Notifications

Performance Records

Core transactional tables remain unpartitioned until operational metrics justify partitioning.

---

# 25. Migration Philosophy

Every migration must be:

Forward compatible

Reviewed

Small

Reversible where possible

Independent

Destructive migrations require explicit planning.

# 28. Golden Rules

The database is the source of truth.

Every table has one responsibility.

Every aggregate owns its data.

Business history is immutable.

Financial correctness is mandatory.

Relationships are explicit.

Business data is normalized.

Indexes exist only for real queries.

Version history is preserved.

Schema evolution must remain backwards compatible.

Every database decision prioritizes long-term maintainability over short-term convenience.

---

# 29. Definition of a Well-Designed Database

A well-designed Strus database should:

Preserve every business rule through explicit constraints.

Maintain complete historical traceability.

Support enterprise-scale workloads without redesign.

Separate business data from infrastructure concerns.

Allow future services and analytics without schema changes.

Remain understandable through consistent naming and ownership.

Serve as the permanent source of truth for every business operation.

---

## Summary

The Database Design Specification defines the persistence layer of Strus. It translates the finalized business architecture and canonical domain model into an implementation-ready PostgreSQL design while preserving data integrity, auditability, scalability, and long-term maintainability. Every table, relationship, constraint, index, and migration defined in subsequent DDS documents must conform to the principles established in this overview.
